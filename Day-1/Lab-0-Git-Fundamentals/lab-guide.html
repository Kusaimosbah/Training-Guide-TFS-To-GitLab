üÜö TFS vs Git: Key Differences

‚ùå TFS/TFVC (Centralized)

  • Check-out/Check-in - Files locked
  • Server-dependent - Need connection
  • Single history - One source of truth
  • Workspace-based - Local mapping
  • Heavyweight branches - Slow operations
  • Shelvesets - Temporary storage

‚úÖ Git (Distributed)

  • No locks - Parallel editing
  • Offline work - Full local repository
  • Distributed - Every clone is complete
  • Repository-based - Git directory
  • Lightweight branches - Fast switching
  • Stash - Local temporary storage
TFS/TFVC Operation Git Equivalent Notes
tf get git pull Get latest changes
tf checkout Not needed Git doesn't lock files
tf checkin git commit + git push Two-step process in Git
tf shelve git stash Temporary save
tf branch git branch / git checkout -b Lightweight in Git
tf merge git merge Similar concept
tf history git log View commit history

Step 1: Configure Your Identity

Set your name and email (this will appear in commits):

# Set global configuration (WSL/Bash) git config --global user.name "Your Name" git config --global user.email "your.email@company.com" # Set default branch name to 'main' git config --global init.defaultBranch main # Set default editor (VS Code) git config --global core.editor "code --wait" # Enable color output git config --global color.ui auto
# Set global configuration (PowerShell) git config --global user.name "Your Name" git config --global user.email "your.email@company.com" git config --global init.defaultBranch main git config --global core.editor "code --wait" git config --global color.ui auto
üí° Configuration Levels:
‚Ä¢ --global - User-level (all repositories)
‚Ä¢ --system - System-level (all users)
‚Ä¢ --local - Repository-level (current repo only)

Step 2: Verify Configuration

# View all configuration git config --list # View specific settings git config user.name git config user.email # View configuration with file locations git config --list --show-origin

Step 3: Configure Line Endings

Important for cross-platform development:

# WSL/Linux - Use LF git config --global core.autocrlf input
# Windows - Convert to CRLF on checkout, LF on commit git config --global core.autocrlf true

üìö Git Basics: The Three Trees

Understanding Git's Three-Tree Architecture

Git uses three "trees" to manage your code:

  1. Working Directory - Your actual files on disk
  2. Staging Area (Index) - Prepared changes for next commit
  3. Repository (.git directory) - Committed history

üéØ Hands-On Practice: Git Commands

Exercise 1: Create Your First Repository

WSL/Bash:

# Create project directory cd ~ mkdir git-fundamentals-lab cd git-fundamentals-lab # Initialize Git repository git init # Check repository status git status # View the .git directory ls -la .git/

PowerShell:

# Create project directory cd ~ mkdir git-fundamentals-lab cd git-fundamentals-lab # Initialize Git repository git init # Check repository status git status # View the .git directory Get-ChildItem -Force .git\
‚úÖ Expected Output:
You should see "Initialized empty Git repository in..." and a .git directory.

Exercise 2: Your First Commit

# Create a README file echo "# Git Fundamentals Lab" > README.md # Check status (file is untracked) git status # Add file to staging area git add README.md # Check status again (file is staged) git status # Commit the staged changes git commit -m "Initial commit: Add README" # View commit history git log # View compact log git log --oneline
üìù Commit Message Best Practices:
  • Use imperative mood: "Add feature" not "Added feature"
  • First line: 50 chars or less (summary)
  • Blank line, then detailed description if needed
  • Reference issue/work item numbers

Exercise 3: Stage Multiple Files

# Create multiple files echo "print('Hello Git')" > app.py echo "*.pyc" > .gitignore mkdir docs echo "# Documentation" > docs/guide.md # Check status git status # Add specific files git add app.py .gitignore # Add entire directory git add docs/ # Check what's staged git status # Commit all staged files git commit -m "Add Python app, gitignore, and documentation"
üîç Common Git Add Patterns:
‚Ä¢ git add filename.txt - Add specific file
‚Ä¢ git add . - Add all changes in current directory
‚Ä¢ git add -A - Add all changes in entire repository
‚Ä¢ git add *.cs - Add all C# files
‚Ä¢ git add -p - Interactive staging (patch mode)

Exercise 4: Viewing Changes

# Modify README echo -e "\n## Getting Started\nThis lab teaches Git basics." >> README.md # View unstaged changes git diff # Stage the change git add README.md # View staged changes git diff --staged # View changes in last commit git show # View specific commit git show HEAD~1
üí° Understanding HEAD:
‚Ä¢ HEAD - Current commit
‚Ä¢ HEAD~1 or HEAD^ - Previous commit
‚Ä¢ HEAD~2 - Two commits ago
‚Ä¢ HEAD^2 - Second parent (for merge commits)

Exercise 5: Undoing Changes

# Scenario 1: Unstage a file (keep changes) echo "temporary content" > temp.txt git add temp.txt git status git reset temp.txt git status # Scenario 2: Discard unstaged changes echo "unwanted change" >> README.md git checkout -- README.md # Scenario 3: Amend last commit echo "# License: MIT" > LICENSE git add LICENSE git commit --amend -m "Add Python app, docs, and license" # Scenario 4: Undo last commit (keep changes) git reset --soft HEAD~1 git status # Scenario 5: Undo last commit (discard changes) - DANGEROUS! git reset --hard HEAD~1
‚ö†Ô∏è Warning:
git reset --hard permanently deletes uncommitted changes. Use with caution!
Always commit or stash important work before using --hard.

Exercise 6: Branching Basics

# View current branches git branch # Create new branch git branch feature/add-calculator # Switch to new branch git checkout feature/add-calculator # Or create and switch in one command git checkout -b feature/add-tests # View all branches with commit info git branch -v # See which branch you're on git status git branch --show-current
‚úÖ Understanding Branches:
In Git, branches are lightweight pointers to commits. Creating a branch is instant!
This is very different from TFS where branching is a heavyweight operation.

Exercise 7: Making Changes on a Branch

# Make sure you're on feature branch git checkout feature/add-calculator # Create a simple calculator file cat > calculator.py << 'EOF' def add(a, b): return a + b def subtract(a, b): return a - b if __name__ == "__main__": print(f"5 + 3 = {add(5, 3)}") print(f"5 - 3 = {subtract(5, 3)}") EOF # Stage and commit git add calculator.py git commit -m "Add calculator with add and subtract functions" # View history with branch visualization git log --oneline --graph --all

Exercise 8: Merging Branches

# Switch back to main branch git checkout main # Verify calculator.py doesn't exist here ls -la # Merge feature branch into main git merge feature/add-calculator # Now calculator.py exists on main ls -la # View commit history git log --oneline --graph --all # Delete the feature branch (optional) git branch -d feature/add-calculator
üìù Merge Types:
‚Ä¢ Fast-forward - Moves pointer forward (no merge commit)
‚Ä¢ Three-way merge - Creates merge commit when branches diverge
‚Ä¢ Squash merge - Combines all commits into one

Exercise 9: Handling Merge Conflicts

# Create two conflicting branches git checkout -b feature/multiply echo "def multiply(a, b):" >> calculator.py echo " return a * b" >> calculator.py git add calculator.py git commit -m "Add multiply function" # Switch to main and create conflict git checkout main echo "def divide(a, b):" >> calculator.py echo " return a / b" >> calculator.py git add calculator.py git commit -m "Add divide function" # Try to merge - will cause conflict git merge feature/multiply # View conflict status git status # View conflicted file cat calculator.py # Open in VS Code to resolve code calculator.py
üîß Resolving Conflicts in VS Code:
  1. VS Code highlights conflicts with colors
  2. Click "Accept Current Change", "Accept Incoming Change", or "Accept Both Changes"
  3. Or manually edit the file to combine changes
  4. Save the file
  5. Stage the resolved file: git add calculator.py
  6. Complete the merge: git commit
# After resolving in VS Code: git add calculator.py git commit -m "Merge feature/multiply with both multiply and divide functions" # View the result git log --oneline --graph --all

Exercise 10: Using Git Stash

# Start making changes echo "# TODO: Add more operations" >> calculator.py # Oops, need to switch branches urgently! git status # Stash your changes git stash save "WIP: Planning calculator enhancements" # Working directory is now clean git status # Do other work, switch branches, etc. git checkout -b hotfix/bug-123 # ... fix bug ... git checkout main # List stashed changes git stash list # Apply stashed changes git stash pop # Or apply without removing from stash git stash apply # Drop specific stash git stash drop stash@{0} # Clear all stashes git stash clear
üí° Git Stash vs TFS Shelvesets:
‚Ä¢ Git stash - Local only, not shared
‚Ä¢ TFS shelveset - Stored on server, can be shared
‚Ä¢ For sharing work in Git, use feature branches instead

Exercise 11: Viewing History

# Basic log git log # Compact one-line format git log --oneline # With graph visualization git log --oneline --graph --all # Last N commits git log -5 # Commits by author git log --author="Your Name" # Commits in date range git log --since="2 weeks ago" --until="yesterday" # Commits affecting specific file git log calculator.py # Show changes in each commit git log -p # Show stats for each commit git log --stat # Pretty formatted log git log --pretty=format:"%h - %an, %ar : %s" # Search commit messages git log --grep="calculator" # Search code changes git log -S "multiply"

Exercise 12: Git Aliases for Productivity

# Create useful aliases git config --global alias.st status git config --global alias.co checkout git config --global alias.br branch git config --global alias.ci commit git config --global alias.unstage 'reset HEAD --' git config --global alias.last 'log -1 HEAD' git config --global alias.visual 'log --oneline --graph --all --decorate' # Now you can use short commands git st git co main git br git visual
‚úÖ Recommended Aliases:
  • git st - Quick status check
  • git co - Fast checkout
  • git visual - Beautiful history graph
  • git unstage - Remove from staging area
