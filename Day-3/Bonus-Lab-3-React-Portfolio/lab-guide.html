‚öôÔ∏è Pipeline Architecture

The pipeline has 4 stages with 5 jobs total:

VALIDATE ‚Üí TEST ‚Üí BUILD ‚Üí DEPLOY

Stage Breakdown

Stage Jobs Purpose
Validate validate:dependencies Check package.json, src/, vite.config.js exist
Test test:dependencies, test:syntax Install npm packages, verify React components
Build build:vite Run npm run build to generate optimized dist/
Deploy pages Copy dist/ to public/, deploy to GitLab Pages

Key Pipeline Features

  • npm caching: Speeds up builds by reusing node_modules
  • Parallel testing: test:dependencies and test:syntax run simultaneously
  • Artifact passing: dist/ from build:vite passed to pages job
  • Main-only deployment: Only main branch deploys to Pages

üéì Challenge Questions

Question Answer
Why use Vite instead of Webpack? Vite is faster: ES modules for dev, pre-bundled deps, faster HMR (Hot Module Reload).
What's in node_modules and why cache it? Contains all npm packages (~500MB). Caching prevents re-downloading on each build, saving 30-60 seconds.
Why separate test and build stages? Fail fast: if tests fail, don't waste time building. Cleaner separation of concerns.
What happens if we deploy from src/ instead of dist/? Browser can't read JSX. React code must be compiled to JavaScript first.
How would we add TypeScript? Vite supports it natively: rename files to .tsx, add tsconfig.json, install TypeScript package.
Can we deploy to a CDN instead of Pages? Yes! Instead of copying to public/, you'd upload dist/ to AWS S3, Netlify, Vercel, etc.
