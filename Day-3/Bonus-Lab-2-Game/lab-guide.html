üîß Understanding the CI/CD Pipeline

Pipeline Architecture

The pipeline consists of 3 stages with 6 jobs total:

VALIDATE ‚Üí TEST ‚Üí DEPLOY

Stage 1: VALIDATE

Job Purpose Checks Image
validate:files Ensure all required files exist ‚úÖ index.html, game.js, style.css present alpine:latest

Stage 2: TEST (3 Parallel Jobs)

Job Purpose Checks Image
test:html Validate HTML5 structure ‚úÖ Valid doctype, title, game container div alpine:latest
test:javascript Validate JavaScript syntax ‚úÖ No syntax errors, Block & TetrisGame classes node:18-alpine
test:css Validate CSS styling ‚úÖ Cell styles, block styles, all 7 colors alpine:latest

Stage 3: DEPLOY

Job Purpose Process Image
pages Deploy to GitLab Pages Copy files to public/, create artifacts, go live alpine:latest
Runner Tag: All jobs have tags: - kusai which means jobs run on a specific runner. In your GitLab instance, only runners with this tag will execute these jobs.

üåê GitLab Pages: A Complete Guide

What is GitLab Pages?

GitLab Pages is a free static site hosting service integrated directly into GitLab:

  • Automatically deploys code from the public/ directory
  • Hosts static files (HTML, CSS, JavaScript, images, etc.)
  • Accessible at a public URL
  • HTTPS enabled by default
  • No server setup required
  • No manual deployment needed

How GitLab Pages Works

Your Commit
    ‚Üì
Push to main branch
    ‚Üì
GitLab CI/CD Pipeline Runs
    ‚Üì
pages job copies files to public/
    ‚Üì
GitLab Pages deploys automatically
    ‚Üì
Game live at: https://your-username.gitlab.io/project-name/

Key Concepts

Concept Explanation
public/ directory All files here are deployed as-is. HTML files served, CSS/JS loaded automatically.
index.html Automatically served as default when visiting the URL.
Pages URL Format: https://{username}.gitlab.io/{project-name}/
Custom Domain Optional: You can configure a custom domain instead of gitlab.io.
SSL/TLS Automatically enabled by default (HTTPS).
CI/CD Requirement Pages job must have artifacts: with public/ path.

üéì Challenge Questions

Question Answer
Why separate test jobs instead of one large test job? Different test types (HTML, JS, CSS) can run in parallel, speeding up the pipeline. Keeps concerns separated.
What if we changed pipeline to run on all branches? Every branch would deploy to Pages, potentially overwriting the main deployment. Not ideal for production.
How could we add a second game to the same project? Create subdirectory structure (e.g., /games/tetris/public/) and configure multiple Pages jobs.
Can we deploy to a custom domain? Yes! Go to Settings ‚Üí Pages and configure a custom domain with DNS records.
What security considerations exist for public Pages? No sensitive data in public/; Pages are accessible worldwide; validate user input in games.

üîß GitLab Pages Server Configuration

For Self-Hosted GitLab Instances: Update your /etc/gitlab/gitlab.rb with the following configuration to enable GitLab Pages:

################################################################################
## GitLab Pages
##! Docs: https://docs.gitlab.com/ee/administration/pages/
################################################################################

##! Define to enable GitLab Pages
pages_external_url "https://pages.your-domain.com"
gitlab_pages['enable'] = true

##! Configure cert when using external IP address
gitlab_pages['cert'] = "/etc/gitlab/ssl/your-domain.com.crt"
gitlab_pages['cert_key'] = "/etc/gitlab/ssl/your-domain.com.key"

##! Optional: Configure HTTP/HTTPS listening ports
# gitlab_pages['external_http'] = ['0.0.0.0:8080']
# gitlab_pages['external_https'] = ['0.0.0.0:8443']

After updating gitlab.rb, run:

sudo gitlab-ctl reconfigure
sudo gitlab-ctl restart gitlab-pages
