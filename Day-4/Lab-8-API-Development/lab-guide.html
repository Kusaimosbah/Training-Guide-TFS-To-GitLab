public class CalculatorService : ICalculatorService
{
    private readonly ILogger<CalculatorService> _logger;
    
    public CalculatorService(ILogger<CalculatorService> logger)
    {
        _logger = logger;
    }
    
    public int Add(int a, int b)
    {
        _logger.LogInformation("Adding {A} + {B}", a, b);
        return a + b;
    }
    
    public double Divide(int a, int b)
    {
        if (b == 0)
            throw new DivideByZeroException("Cannot divide by zero");
        return (double)a / b;
    }
    // ... more operations
}

Example CalcController structure:

[ApiController]
[Route("api/[controller]")]
public class CalcController : ControllerBase
{
    private readonly ICalculatorService _calculatorService;
    
    public CalcController(ICalculatorService calculatorService)
    {
        _calculatorService = calculatorService;  // Injected!
    }
    
    [HttpGet("add")]
    public IActionResult Add([FromQuery] int a, [FromQuery] int b)
    {
        try
        {
            var result = _calculatorService.Add(a, b);
            return Ok(new { operation = "add", a, b, result });
        }
        catch (Exception ex)
        {
            return BadRequest(new { error = ex.Message });
        }
    }
    
    [HttpGet("divide")]
    public IActionResult Divide([FromQuery] int a, [FromQuery] int b)
    {
        try
        {
            var result = _calculatorService.Divide(a, b);
            return Ok(new { operation = "divide", a, b, result });
        }
        catch (DivideByZeroException ex)
        {
            return BadRequest(new { error = ex.Message });
        }
    }
}

Key Concepts:

  • [ApiController] - Enables automatic model validation, binding
  • [Route("api/[controller]")] - Creates /api/calc route
  • [HttpGet("add")] - Creates /api/calc/add endpoint
  • [FromQuery] - Gets parameters from URL query string (?a=5&b=3)
  • IActionResult - Flexible return type (Ok, BadRequest, NotFound, etc.)
  • Constructor injection - Service is injected automatically

13Understand Data Flow

üìä Request Flow for Calculator API:
  1. HTTP Request ‚Üí GET /api/calc/add?a=5&b=3
  2. CalcController ‚Üí Receives request, extracts query parameters
  3. CalcController ‚Üí Calls _calculatorService.Add(5, 3)
  4. CalculatorService ‚Üí Executes business logic (5 + 3 = 8)
  5. CalculatorService ‚Üí Logs operation, returns result (8)
  6. CalcController ‚Üí Formats HTTP response: Ok(new { operation, a, b, result })
  7. HTTP Response ‚Üí 200 OK with JSON: {"operation": "add", "a": 5, "b": 3, "result": 8}

14Understand Dependency Injection Benefits

‚úÖ Why DI is Powerful in CapstoneApi:
  • Testability: Easy to mock CalculatorService for unit tests (as you saw in tests/)
  • Flexibility: Can swap CalculatorService with different implementation
  • Maintainability: Single responsibility - CalcController handles HTTP, CalculatorService handles math
  • Decoupling: CalcController doesn't create CalculatorService - it's injected
  • Logging: ILogger is also injected, enabling structured logging throughout

üîß Troubleshooting

API Won't Start

# Ensure .NET 8 SDK is installed
dotnet --version

# Clean and rebuild
cd src/CapstoneApi
dotnet clean
dotnet build
dotnet run

Tests Failing Locally

# Clean test artifacts
cd tests/CapstoneApi.Tests
dotnet clean
dotnet build

# Run tests with verbose output
dotnet test --verbosity normal

Swagger Not Loading

  • Check API is running: Look for "Now listening on: http://localhost:5000"
  • Verify Swagger URL: http://localhost:5000/swagger (not /swagger/index.html)
  • Check Program.cs has app.UseSwagger() and app.UseSwaggerUI()

Integration Tests Failing

  • Ensure Program.cs has public partial class Program { } at the end
  • This is required for WebApplicationFactory to access Program class
  • Already fixed in starter code!

Pipeline Job Failed

  • Click on the failed job in GitLab CI/CD to view full logs
  • Check build/test logs for specific error messages
  • Ensure all code changes are committed before pushing

Can't Access Deployed API

  • Verify CI/CD variables are set: DEPLOY_USER and DEPLOY_PASSWORD
  • Check deploy stage logs for SSH errors
  • Verify Ubuntu server is running: ping 139.59.248.22
  • See DEPLOYMENT-SETUP.md for complete deployment troubleshooting
