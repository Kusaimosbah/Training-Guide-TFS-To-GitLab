4Examine Integration Tests

cat tests/CapstoneApi.Tests/Integration/CalcControllerTests.cs

Integration tests use WebApplicationFactory to test HTTP endpoints:

using System.Net;
using Microsoft.AspNetCore.Mvc.Testing;
using Xunit;

namespace CapstoneApi.Tests.Integration;

public class CalcControllerTests : IClassFixture<WebApplicationFactory<Program>>
{
    private readonly HttpClient _client;

    public CalcControllerTests(WebApplicationFactory<Program> factory)
    {
        _client = factory.CreateClient();  // Creates test HTTP client
    }

    [Fact]
    public async Task Add_TwoNumbers_ReturnsCorrectResult()
    {
        // Act - Make HTTP GET request
        var response = await _client.GetAsync("/api/calc/add?a=5&b=3");

        // Assert - Check status and response
        Assert.Equal(HttpStatusCode.OK, response.StatusCode);
        var content = await response.Content.ReadAsStringAsync();
        Assert.Contains("\"result\":8", content);
    }
    
    [Fact]
    public async Task Divide_ByZero_ReturnsBadRequest()
    {
        // Act
        var response = await _client.GetAsync("/api/calc/divide?a=10&b=0");
        
        // Assert
        Assert.Equal(HttpStatusCode.BadRequest, response.StatusCode);
    }
}
üìå Integration Test Characteristics:
  • Tests full HTTP request/response cycle
  • API starts in-memory (WebApplicationFactory)
  • Tests routing, controllers, services together
  • Slower than unit tests (100ms+ per test)
  • More realistic - tests what users experience

5Understand WebApplicationFactory

‚úÖ What WebApplicationFactory Does:
  • Starts your API in-memory (no real server needed!)
  • Creates an HttpClient to make requests
  • Runs the ENTIRE API stack (controllers, services, middleware)
  • Tests run isolated - no conflicts between tests
  • Requires public partial class Program { } in Program.cs (already done!)
üìå IClassFixture: Ensures WebApplicationFactory is created once per test class for performance.

9Understand Test Filtering in CI/CD

Open .gitlab-ci.yml and see how tests are separated:

unit-test:
  stage: test
  script:
    - dotnet test --filter "FullyQualifiedName~Unit"
  
integration-test:
  stage: test
  script:
    - dotnet test --filter "FullyQualifiedName~Integration"
üìå Why Separate in CI/CD?
  • Parallel execution - both stages run at same time
  • Faster feedback - unit tests finish quickly
  • Better diagnostics - know which type failed
  • Independent reporting - separate test results

10Compare Unit vs Integration Tests

Aspect Unit Tests Integration Tests

Speed Very fast (ms) Slower (100ms+)
Dependencies Mocked/isolated Real (in-memory)
What's Tested Business logic HTTP behavior, routing, serialization
Example CalculatorService.Add(5, 3) GET /api/calc/add?a=5&b=3 HTTP request

11Understand When to Use Each

üéØ Testing Strategy for CapstoneApi:
  • Unit Tests: Test CalculatorService methods (Add, Subtract, Divide, etc.)
  • Integration Tests: Test API endpoints (/api/calc/add, /api/info, /api/status/health)
  • Both Together: Comprehensive coverage - fast feedback + realistic behavior!

Typical Ratio: 70% unit tests, 30% integration tests

CapstoneApi follows this: ~8 unit tests, ~7 integration tests
